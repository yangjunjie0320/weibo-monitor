# 打分与锐评沉淀机制 — 已归档设计草案

> 此方案已被 2026-07 的“单按钮人工转发 + 多维表格归档”流程取代，本文仅保留
> 历史背景，不代表当前实现。当前行为以 README、`src/forward.py` 和
> `config.example.yaml` 为准。

> 对应群里的需求："呈现 → 快速打分 + 锐评 → 打分沉淀成对账号打分
>（比如 5 分内容 20%、3 分内容 70%，判定为 T1 账号）+ 锐评沉淀成老马的观点价值判断"

## 目标

1. 编辑在群里对每张卡片**一键打分**（好/中/差），顺手可以**回复锐评**
2. 分数按博主聚合，形成**账号分层**（T1/T2/T3），反哺推送策略
3. 锐评按博主/主题归档，沉淀成可检索的观点库

## 交互设计

### 打分：卡片按钮

卡片底部加一排按钮：`[5 好料] [3 一般] [1 没用] [打开原帖]`。
三档比五档快——打分动作要在 1 秒内完成，档位多了没人用。

点击后机器人**原地更新卡片**（飞书 patch message 接口），把按钮行替换成
"已评：5 分（老马）"，防止重复打分、也让其他人看到评价。

### 锐评：回复卡片

对卡片消息**直接回复文字**即视为锐评（不需要 @机器人）。机器人监听消息事件，
凡 parent_id 是自己发的卡片的回复都归档，关联到对应的 mid/博主。

## 技术方案

### 需要新增的基础设施：长连接监听

现在机器人只发不收。打分回调和回复监听都要求事件订阅，复用 feishu-link
的成熟模式（`lark_oapi.ws.Client`，无需公网回调）：

- `card.action.trigger`：卡片按钮点击（打分）
- `im.message.receive_v1`：群消息（筛出对卡片的回复 → 锐评）

新增 `src/listener.py`（照搬 feishu-link listener 裁剪），在 main 的
asyncio 事件循环里与轮询监控并行跑。飞书应用需追加权限：
`im:message.receive_v1`（接收消息事件）+ 卡片回调无需额外权限。

### 数据存储：SQLite

state/ratings.db，两张表（JSON 文件不适合多维查询了）：

```sql
ratings(mid TEXT, uid TEXT, score INT, rater TEXT, created_at TEXT,
        PRIMARY KEY (mid, rater))
comments(id INTEGER PRIMARY KEY, mid TEXT, uid TEXT, rater TEXT,
         text TEXT, created_at TEXT)
```

发卡时需记住 message_id ↔ mid 的映射（新表 messages(message_id, mid, uid)），
否则回调/回复无法关联回帖子。

### 账号分层

每周（或每 N 条评分后）计算每个 uid 的分数分布：

- T1：均分 ≥ 4 且样本 ≥ 10 —— 推送时标题加 ★，永不折叠
- T2：默认
- T3：均分 ≤ 2 且样本 ≥ 10 —— 整卡折叠（同广告处理），连续低分可提醒移出监控

分层结果写 state/tiers.json，推送时读取；每周日往群里发一张「账号周报」卡片
（各账号评分分布 + 本周锐评摘录），分层调整让老马确认而不是全自动。

## 实施步骤（预估 1-2 天）

1. listener.py + 事件循环并行改造，`--listen` 开关（半天）
2. ratings.db + 发卡记 message_id + 打分按钮 + 卡片原地更新（半天）
3. 回复归档锐评（2 小时）
4. 分层计算 + 周报卡片 + 推送策略挂钩（半天）

## 风险与取舍

- **长连接一个 app 只能一个进程**：监控和监听必须在同一进程内（asyncio 并行），
  不能拆两个服务。
- **卡片 schema 2.0 的按钮回调**：需要用 `behaviors: [{"type": "callback"}]`
  形式，与现在的 open_url 按钮并存；老卡片（已发出的）没有打分按钮，不回溯。
- **谁能打分**：默认群里所有人；如需只认老马，按 open_id 过滤（配置项）。
- 锐评→"观点价值判断"的进一步提炼（LLM 汇总老马的观点倾向）先不做，
  等锐评攒够一个月再说。
